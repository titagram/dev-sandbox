# DevBoard Frontend And Quality Control Layer Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Redesign DevBoard's frontend as a complete API-backed operational dashboard while preserving a later, incremental Laravel Quality & Test Control Layer as a first-class product surface.

**Architecture:** Split the work into two independent tracks. emergent.sh should build the frontend experience, typed API adapter, mock data, and future Quality Center screens only; Codex/Laravel implementation should later add real backend quality commands, registries, reports, and gates under the existing Laravel app. The frontend must not depend on Symfony-style command names or plugin-only `/api/plugin/v1` endpoints.

**Tech Stack:** Laravel 13, Inertia/React current backend, future standalone React/Vite frontend from emergent.sh, Pest/PHPUnit, Composer audit, Laravel Pint, optional PHPStan, Playwright, ZAP/Semgrep/Trivy/Nuclei as opt-in scanners.

## Current-State Considerations

- `verified_from_code`: the backend is Laravel 13, not Symfony. Use Artisan commands such as `php artisan quality:route-inventory`, not `php bin/console app:quality:route-inventory`.
- `verified_from_code`: backend tests are Pest/PHPUnit under `backend/tests`.
- `verified_from_code`: frontend build is Vite/React under `backend/resources/js`.
- `verified_from_code`: `backend/composer.json` includes Pest, PHPUnit, Laravel Pint, Laravel Sanctum, and Inertia; it does not declare PHPStan as a direct dev dependency even though transitive package metadata mentions it.
- `verified_from_code`: existing dashboard routes are Laravel web/Inertia routes; plugin endpoints under `/api/plugin/v1` are for local CLI/MCP plugin use, not browser UI.
- `developer_provided`: the emergent.sh frontend source will be pulled from `https://github.com/titagram/emergent_devboard_frontend.git`, and the AI worker must read that repository's `app/README_LLM.md` before integrating or modifying the generated frontend.
- `developer_provided`: the desired quality system must be deterministic, incremental, safe for AI-assisted workflows, and oriented around domain truth rather than simple "route returns 200" checks.
- `developer_provided`: destructive scans, active DAST, production scans, payment tests, email sends, and destructive route calls must never run by default.
- `inferred`: because this repository already has `ai-sandbox/config`, quality backend configuration should live under `backend/config/quality/` to avoid root-level config ambiguity.
- `needs_verification`: before implementing YAML parsing, confirm whether `symfony/yaml` is available as a runtime dependency or add it explicitly to `backend/composer.json`.

## Backend Quality Layer Scope

In scope for the later Laravel implementation:

- documentation under `docs/quality-system/`;
- Laravel-local config under `backend/config/quality/`;
- generated reports under `backend/var/quality/reports/`;
- Laravel Artisan commands under `backend/app/Console/Commands/Quality/`;
- service classes under `backend/app/Quality/`;
- Pest feature tests under `backend/tests/Feature/Quality/`;
- Composer scripts in `backend/composer.json`;
- optional Playwright scaffolding in `backend/tests/e2e/`.

Out of scope for the first backend slice:

- monolithic custom scanner engines;
- active scans against production;
- complete browser suite;
- complete authorization matrix;
- complete mutation/load/security scanner rollout;
- changing existing business logic to satisfy quality checks.

## Functional Task Distribution Check

The work is intentionally split into aggregates that can be executed and reviewed independently:

1. **Frontend source intake, AI-owned:** pull `titagram/emergent_devboard_frontend`, read `app/README_LLM.md`, inspect dependency/build/test instructions, and verify the generated frontend before integration.
2. **Frontend track, emergent.sh-owned:** UI, navigation, typed models, API adapter, mock data, and Quality Center screens. It must not implement Laravel backend commands or scanners.
3. **Backend quality documentation and registries:** documents and config files only. This creates shared language before code.
4. **Backend quality report primitives:** report data model and writers. Route and gate commands depend on this.
5. **Backend route inventory:** read-only route extraction. This is safe and should land before route smoke.
6. **Backend route smoke:** safe execution of configured `SAFE_READ` routes. This depends on registries and report primitives.
7. **Backend quality gates:** consumes reports and determines pass/fail. This depends on report primitives and route smoke outputs.
8. **Dependency/tooling integration:** Composer scripts, PHPStan/Pint/audit wiring, and optional tool installation. This happens after command contracts are stable enough to verify.
9. **Browser/security scanner preparation:** documentation and disabled-by-default optional integration. This stays separate from the MVP route/gate layer to avoid unsafe or heavy default runs.
10. **Future dashboard API:** read-only JSON endpoints first, write/run endpoints later behind permissions and approvals.

This distribution is functional because each aggregate has a clear dependency direction and a bounded verification command. Avoid merging frontend generation with backend scanner implementation, and avoid enabling optional scanners before the route/report/gate MVP exists.
