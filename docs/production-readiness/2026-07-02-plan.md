# Production Readiness Plan - DevBoard - 2026-07-02

## Executive Summary

DevBoard is close to a local self-hosted launch candidate, but it is not yet production-ready. The strongest foundations are already present: Laravel dashboard and API routes, plugin/Hades token authentication, PostgreSQL and Neo4j runtime services, artifact storage, quality reports, backup export/dry-run validation, and a separate React dashboard frontend wired through a typed adapter.

The launch blockers are operational, integration, and release-discipline issues rather than a missing core concept:

- P0: freeze and reconcile the external frontend release state in `/home/ubuntu/emergent_devboard_frontend`.
- P0: replace dev-oriented container startup and default secrets with production deployment artifacts.
- P0: add a real browser acceptance path for `REACT_APP_USE_MOCK=false`.
- P0: finish a verified backup/restore story, not only export and dry-run validation.
- P0: add CI/release gates that run backend, plugin, analyzer, frontend, and Docker checks.

Recommended sequence: stabilize the frontend baseline first, then harden deployment/env, then prove a full HTTP-adapter browser flow, then add restore validation and CI. Security, observability, performance, accessibility, and documentation become meaningful after those foundations are repeatable.

## Risks And Decisions

- Risk: external frontend dirty state hides whether production code is reviewed and reproducible.
- Risk: current Docker startup installs dependencies at runtime, which can make deploys slow, flaky, and network-dependent.
- Risk: default local secrets in compose are unsafe if reused outside local development.
- Risk: backup export without verified restore can create false confidence.
- Risk: no CI means regressions across backend, frontend, plugin, analyzer, and agent boundaries can ship unnoticed.
- Risk: production observability is not enough to detect queue, graph, storage, backup, or frontend failures proactively.
- Decision needed: active browser shell and deprecation/removal plan for Laravel Inertia dashboard.
- Decision needed: final domain, ingress, Basic Auth posture, and TLS management.
- Decision needed: RPO/RTO, backup storage location, and restore strategy.
- Decision needed: privacy/retention policy for source artifacts, prompts, audit logs, backups, and agent memory.
- Decision needed: support and incident ownership.
