# Production Readiness Plan - DevBoard Agent And Memory Follow-Up - 2026-07-03

## Executive Summary

This plan covers the concrete production issues reported on `https://home-sweet-home.cloud`:

- `POST /api/dashboard/tasks/{task}/assistant/clarify` returns HTTP 500.
- Task detail pages need a faster status change control.
- Agent work items and agent chat threads need user-facing delete/archive actions.
- Project Memory table overflow hides the `Actions` column.
- Memory import queue rows need cancelability, detail inspection, and clearer status semantics.
- Manual memory needs a `logbook` kind, but long logbooks need a dedicated ingestion model rather than being treated as ordinary short memory notes.
- Server and local agents need a clearer operating model.

The main live blocker is the Task Clarifier 500. Runtime logs show the configured provider is an OpenAI-compatible chat-completions endpoint, while `TaskClarifierService` uses the Laravel AI SDK OpenAI gateway path that posts to `/responses`. The remote provider returns 404, and the exception currently escapes as HTTP 500. The same provider-use pattern exists in Backlog Triage, so the fix should be shared instead of patched only for one route.

Recommended order:

1. Fix AI provider routing/error handling for Task Clarifier and Backlog Triage.
2. Add the quick task status control.
3. Add soft delete/archive for Agent Work and Agent Chat.
4. Fix Memory UI overflow and import details/cancel/status wording.
5. Add `logbook` as a manual kind, then implement a dedicated logbook ingestion surface for long entries.
6. Align agent roles, registry records, UI labels, and allowed tools.

### P0-1: Stop Task Clarifier And Backlog Triage From Returning 500 On Provider Failure

- Area: Backend, AI provider integration
- Objective: Make assistant endpoints return a useful suggestion or a controlled API error instead of HTTP 500.
- Evidence:
  - `POST /api/dashboard/tasks/01KWF9JMN67B7XDFCQJE5CTHJE/assistant/clarify` produced production 500.
  - `backend/storage/logs/laravel.log` inside `devboard-app-1` shows `HTTP request returned status code 404` from the Laravel AI OpenAI gateway while handling the clarify route.
  - DB registry currently has `provider_type=openai_compatible`, `provider_key=openai`, `base_url=https://opencode.ai/zen/go/v1/chat/completions`, model `deepseek-v4-flash`.
  - `TaskClarifierService::configureLaravelAiProvider()` sets the Laravel AI provider `url` to that base URL.
  - Laravel AI SDK OpenAI gateway posts to a responses-style endpoint, while `ServerAgentWorkService` normalizes the same base URL to `/chat/completions` and succeeds/fails with controlled work item status.
  - `BacklogTriageService` has the same SDK/provider pattern.
- Affected paths:
  - `backend/app/Assistants/TaskClarifierService.php`
  - `backend/app/Assistants/BacklogTriageService.php`
  - `backend/app/Assistants/Agents/TaskClarifierAgent.php`
  - `backend/app/Assistants/Agents/BacklogTriageAgent.php`
  - `backend/app/Services/ServerAgentWorkService.php`
  - `backend/database/seeders/DevBoardSeeder.php`
  - `backend/tests/Feature/Dashboard/TaskClarifierDashboardTest.php`
  - `backend/tests/Feature/Dashboard/BacklogTriageDashboardTest.php`
  - `backend/tests/Feature/Dashboard/AgentWorkDashboardApiTest.php`
- What to change:
  - Introduce one provider execution adapter used by Task Clarifier, Backlog Triage, and server agent work where possible.
  - Decide per provider whether to use Laravel AI SDK responses flow or chat-completions flow.
  - Validate/admin-label provider base URLs so a chat-completions endpoint is not accidentally sent into a responses gateway.
  - Catch provider HTTP exceptions and convert them into deterministic fallback suggestions or a clear 502/assistant failure payload.
  - Persist failed assistant runs with metadata when a real provider call fails, so the UI can explain the failure without losing auditability.
- Recommended implementation:
  - Add an `AiProviderRuntime` or similar service that exposes `structuredAgentPrompt()` and `chatCompletion()` entrypoints.
  - For `openai_compatible` providers with base URL ending in `/chat/completions`, use chat-completions directly or normalize the base URL consistently.
  - Keep Laravel AI SDK fake support for tests.
  - Add `execution_mode=provider_failed_fallback` when deterministic fallback is returned after provider failure.
  - Redact provider URL/API key details in browser responses and logs.
- Acceptance criteria:
  - The clarify route no longer returns 500 for the reported task.
  - Provider 404/timeout/auth errors produce either deterministic suggestion output or a controlled JSON error with status 502.
  - Backlog triage has the same failure behavior.
  - Assistant run/message/audit rows clearly identify fake, deterministic fallback, provider success, and provider failure fallback.
- Verification:
  - `cd backend && php artisan test tests/Feature/Dashboard/TaskClarifierDashboardTest.php`
  - `cd backend && php artisan test tests/Feature/Dashboard/BacklogTriageDashboardTest.php`

### P1-8: Create A Dedicated Long-Logbook Ingestion Model

- Area: Product architecture, memory, agents
- Objective: Support long operational logs without stuffing huge documents into ordinary memory entries.
- Evidence:
  - Existing docs describe logbook as first-class shared operational memory.
  - Current `project_memory_entries` is good for bounded summaries and searchable memory, not raw long-running logs.
  - The user expects local agents to push logbook updates over time.
- Affected paths:
  - New backend migrations for logbook sources/chunks, or carefully extended `project_memory_entries`.
  - `backend/app/Http/Controllers/Plugin/*`
  - `backend/app/Http/Controllers/Dashboard/Api/*Memory*`
  - `backend/app/Assistants/Tools/SearchProjectMemoryTool.php`
  - `backend/app/Services/MemoryImportService.php`
  - Hades/plugin/local agent endpoints and docs.
  - `/home/ubuntu/emergent_devboard_frontend/frontend/src/pages/ProjectMemoryPage.tsx`
  - A possible future `/projects/{project}/logbook` route.
- Recommendation:
  - Treat logbook as an append-only source stream plus summarized memory entries.
  - Store raw or semi-raw long content in `project_logbook_sources` and `project_logbook_chunks` with hashes, offsets, actor/source, repository/task/run links, and ingestion metadata.
  - Create concise `project_memory_entries` summaries with `kind=logbook` for retrieval, search, and agent context.
  - Let local agents push deltas, not the whole logbook every time.
  - Use internal agents only for summarization/classification/extraction, not as the sole source of truth.
  - Keep chunking deterministic and idempotent so retries do not duplicate entries.
- Proposed lifecycle:
  - `local_agent pushes delta`
  - `backend stores chunk with content hash`
  - `Watchman or a logbook summarizer creates bounded memory summaries`
  - `Socrates/Watchman search summaries first`
  - `detail view opens raw chunk/provenance when needed`
- Acceptance criteria:
  - Long logs can be ingested without request timeouts or oversized prompt/context payloads.
  - Each summary links back to raw chunks and evidence.
  - Re-sending the same delta is idempotent.
  - Agents can search logbook summaries and fetch bounded excerpts.
  - Retention/export policy is documented.
- Verification:
  - Backend tests for chunk idempotency, pagination, summary creation, and search.
  - Plugin/Hades integration test for append/fetch delta.
  - Browser test for logbook detail view and summary list.

### P1-9: Organize Agents Around Explicit Product Roles

- Area: Product architecture, AI registry, frontend UX
- Objective: Make each agent understandable and reduce overlap between server specialists, local execution, and memory monitoring.
- Evidence:
  - Seeded profiles are `socrate_supervisor`, `task_clarifier`, `backlog_triage`, `wiki_query`, and `watchman`.
  - Frontend Agent Work/Chat exposes `socrates`, `platon`, `aristoteles`, and `local_agent`.
  - `ServerAgentWorkService` maps `socrates => socrate_supervisor`, `platon => task_clarifier`, `aristoteles => backlog_triage`.
  - `watchman` has seeded allowed tools such as `read_logbook_entries`, but those tools are not all implemented in the current registry.
- Recommended model:
  - `Socrates`: primary project interface. Answers project-scoped questions, cites memory/wiki/tasks/runs/evidence, and routes to specialists. Read-only by default.
  - `Platon`: task clarity specialist. Works from a task, asks concrete clarification questions, produces acceptance criteria and risk/test hints.
  - `Aristoteles`: backlog/process analysis specialist. Looks for bottlenecks, vague tasks, recurring failures, and prioritization risks.
  - `Wiki`: documentation/evidence specialist. Reads wiki pages/revisions, freshness, conflicts, and suggests doc updates.
  - `Watchman`: memory/logbook/risk monitor. Reacts to logbook entries, run failures, quality reports, stale docs, and local-agent completions.
  - `Local Agent`: execution worker near the repository. Claims approved work items, runs code/tests/scans locally, uploads evidence, and appends logbook deltas.
- What to change:
  - Update seeded agent descriptions, frontend labels, and help text to match the model.
  - Add agent capability metadata so the UI can show what an agent can read, suggest, and mutate.
  - Remove or hide allowed tools that are not implemented, or implement the missing tools before enabling them.
  - Prefer `Socrates` as the default chat agent and expose specialist selection as an intentional choice.
  - Keep server agents from executing repository operations directly.
- Acceptance criteria:
  - Agent selection in Agent Work and Agent Chat tells users which agent to pick.
  - Registry seed data, admin UI, frontend labels, and docs use the same names and responsibilities.
  - Disabled/unimplemented agent capabilities are not advertised as live.
