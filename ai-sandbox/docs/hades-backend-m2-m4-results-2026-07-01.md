# Hades Backend M2-M4 Results - 2026-07-01

Remote backend workspace: `/home/ubuntu/dev-sandbox/backend`.

Status: M2, M3 and M4 are implemented on top of the existing M1 Hades API.
The implementation keeps the Hades API under `/api/hades/v1`, uses bearer
agent tokens for operational routes, and keeps `agent_id` from Hades as an
external id while Laravel owns the backend primary keys.

## M2 Workspace Binding

Implemented routes:

- `POST /api/hades/v1/workspaces/bind`
- `POST /api/hades/v1/workspaces/{workspaceBinding}/unlink`

Implemented storage:

- `hades_workspace_bindings`

Contract decisions:

- `workspace_binding_id` is generated by Laravel as an ULID.
- The client sends `project_id`, `agent_id`, `workspace_fingerprint`, optional
  `local_project_id`, redacted `display_path`, git remote display/hash and
  `head_commit`.
- No raw local repo path is required or stored by the API contract.
- Binding is idempotent for `project_id + hades_agent_id + workspace_fingerprint`.
- If a linked binding with the same fingerprint exists for another project,
  the API returns `409 workspace_project_conflict`.
- Unlink marks the binding `unlinked` and preserves backend history.

## M3 Shared Memory

Implemented routes:

- `GET /api/hades/v1/memory/snapshot`
- `POST /api/hades/v1/memory/proposals`

Implemented storage:

- `hades_memory_proposals`
- Existing `project_memory_entries` remains the authoritative shared-memory
  source.

Contract decisions:

- Snapshot returns `version`, `snapshot_version`, `etag` and `items`.
- Items expose memory id, source, kind, summary, decoded JSON payload,
  timestamps and per-entry version.
- Proposal create/update/delete is accepted through one endpoint.
- Low-risk `create` proposals are auto-accepted for the MVP and create a
  `project_memory_entries` row with `source=hades_agent` and `kind=proposal`.
- Proposal idempotency uses `workspace_binding_id + local_proposal_id`.
- Memory access against an unlinked binding returns
  `409 workspace_binding_unlinked`.

## M4 Agent Jobs

Implemented routes:

- `GET /api/hades/v1/agent/jobs`
- `POST /api/hades/v1/agent/jobs/{job}/status`
- `POST /api/hades/v1/agent/jobs/{job}/result`

Implemented storage:

- `hades_agent_jobs`
- `hades_agent_job_events`

Contract decisions:

- Pull returns `queued` jobs scoped by project, external agent, workspace
  binding and requested `capabilities[]`.
- Supported lifecycle spelling is `received`, `waiting_confirmation`,
  `started`, `completed`, `failed`, `expired`, `cancelled`, `unlinked`.
- Status and result submissions create event rows.
- Result payload is structured JSON. Hades client remains responsible for
  local bounding/redaction before submit.
- Workspace unlink marks non-terminal jobs for that binding as `unlinked` and
  leaves terminal jobs unchanged.
